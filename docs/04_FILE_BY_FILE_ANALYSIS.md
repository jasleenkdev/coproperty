# CoProperty — File-by-File Detailed Analysis

Every file in the project, explained line-by-line where it matters.

---

## Backend Files

### `backend/core/core/settings.py`
- Standard Django 5.2 settings generated by `django-admin startproject`
- **Key configurations**:
  - `INSTALLED_APPS`: `rest_framework` (DRF for REST API), `properties` (main app), `corsheaders` (CORS)
  - `MIDDLEWARE`: `CorsMiddleware` placed first (correct position)
  - `DATABASES`: SQLite3 at `BASE_DIR / db.sqlite3`
  - `CORS_ALLOW_ALL_ORIGINS = True` — allows any frontend origin (dev convenience, security risk in prod)
  - `DEBUG = True` — development mode
  - `ALLOWED_HOSTS = []` — only works in debug mode

### `backend/core/core/urls.py`
- **Bug**: `urlpatterns` defined twice — second overwrites first
- Routes:
  - `/admin/` → Django admin panel
  - `/` → includes `properties.urls`

### `backend/core/properties/models.py` — **The Heart of the System**

Five models define the entire domain:

| Model | Purpose | Key Fields |
|-------|---------|------------|
| **Property** | A real-estate asset | `name`, `location`, `purchase_price`, `monthly_rent`, `maintenance_cost` |
| **Ownership** | Token ownership record | `user` (FK→User), `property` (FK→Property), `tokens_owned` |
| **RentPayout** | Monthly rent distribution record | `user`, `property`, `amount`, `month` |
| **Proposal** | Governance proposal | `property`, `title`, `proposal_type`, `description`, `is_executed` |
| **Vote** | Individual vote on a proposal | `proposal`, `user`, `vote` (bool), `tokens_used` |

**Business methods**:
- `Property.annual_roi()`: `((monthly_rent - maintenance_cost) × 12 / purchase_price) × 100`
- `Ownership.ownership_percentage()`: `(tokens_owned / 1000) × 100`

### `backend/core/properties/views.py`
- All views are **function-based** using `@api_view` decorator
- 7 endpoint functions (see API table in Architecture doc)
- `property_proposals` builds a custom response by calling `proposal_result()` for each proposal
- `vote_on_proposal` has the insecure `request.user or User.objects.first()` fallback

### `backend/core/properties/serializers.py`
- **PropertySerializer**: Includes computed `roi` field via `SerializerMethodField`
- **OwnershipSerializer**: Includes computed `ownership_percentage` field
- **RentPayoutSerializer**: Straightforward field mapping

### `backend/core/properties/services.py`
- **`distribute_rent(property_obj)`**: Calculates net rent, iterates over ownerships, creates payouts using `get_or_create` (idempotent — won't double-pay)
- **`cast_vote(proposal, user, vote_choice)`**: Looks up user's ownership in the property, creates Vote with their token count as voting weight
- **`proposal_result(proposal)`**: Sums tokens FOR and AGAINST, returns approval status

### `backend/core/properties/admin.py`
- Rich admin panel with computed columns:
  - PropertyAdmin: shows `annual_roi_display` column
  - OwnershipAdmin: shows `ownership_percentage_display` column
  - RentPayoutAdmin, ProposalAdmin, VoteAdmin: standard

### `backend/core/properties/management/commands/distribute_rent.py`
- Django management command to run monthly rent distribution
- Iterates over all Properties, calls `distribute_rent()` for each
- Must be run manually: `python manage.py distribute_rent`

### `backend/core/properties/migrations/`
- 5 migration files showing the evolution:
  1. `0001_initial.py` — Property model
  2. `0002_ownership.py` — Ownership model
  3. `0003_rentpayout.py` — RentPayout model
  4. `0004_alter_rentpayout_month_and_more.py` — Added month uniqueness
  5. `0005_proposal_vote.py` — Governance models

## Frontend Files

### `frontend/app/src/App.js`
- Minimal: just renders `<Properties />` component
- No routing, no global state, no layout wrapper

### `frontend/app/src/api/api.js`
- 5 exported functions:
  - `getProperties()` — GET /properties/
  - `getPropertyOwnership(id)` — GET /properties/{id}/ownership/
  - `getPropertyPayouts(id)` — GET /properties/{id}/payouts/
  - `getPropertyProposals(id)` — GET /properties/{id}/proposals/
  - `voteOnProposal(proposalId, vote)` — POST /proposals/{id}/vote/

### `frontend/app/src/pages/properties.js`
- Renders a list of clickable property cards showing name, location, ROI
- Clicking a card sets `selectedProperty`, which conditionally renders `PropertyDetail`
- Back button sets `selectedProperty` to null

### `frontend/app/src/pages/PropertyDetail.js`
- Three sections:
  1. **Ownership Distribution** — table of users, tokens, ownership %
  2. **Rent Payout History** — table of users, amounts, dates
  3. **Governance Proposals** — cards with vote counts, FOR/AGAINST buttons
- Data fetched via `useEffect` on `property.id` change

### `frontend/app/src/components/OwnershipPie.js`
- ⚠️ **UNUSED** — built but never imported into any page
- Recharts PieChart showing ownership distribution
- Takes `ownerships` array as prop

### `data/properties.json`
- 2 sample properties (Bangalore, Pune) — purely reference data
- No automated import mechanism exists
